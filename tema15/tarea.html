API de Sistema de Reservas de Restaurante

1. Descripción del Proyecto

Desarrollarás una API REST completa para gestionar el sistema de reservas de un restaurante llamado "La Mesa Dorada". El restaurante necesita modernizar su sistema de reservas, actualmente basado en papel, y ha contratado tus servicios para crear una solución digital.

La API debe permitir gestionar clientes, mesas y reservas, garantizando que no se produzcan conflictos de horarios y proporcionando estadísticas de ocupación para ayudar a la gerencia en la toma de decisiones.

2. Contexto del Negocio

El restaurante "La Mesa Dorada" cuenta con las siguientes características:

  • Horario de operación: 12:00 a 16:00 (comidas) y 20:00 a 00:00 (cenas)
  • Capacidad: 15 mesas de diferentes tamaños (2, 4, 6 y 8 personas)
  • Duración estándar de reserva: 2 horas
  • Política de cancelación: Las reservas pueden cancelarse hasta 2 horas antes

3. Requisitos Funcionales

3.1. Gestión de Clientes

Endpoint Método Descripción
/clientes/ GET Listar todos los clientes con paginación
/clientes/{id} GET Obtener un cliente por ID
/clientes/ POST Crear un nuevo cliente
/clientes/{id} PUT Actualizar datos de un cliente
/clientes/{id} DELETE Eliminar un cliente (solo si no tiene reservas activas)
/clientes/buscar/ GET Buscar clientes por nombre, email o teléfono

Datos del cliente:

  • ID (autogenerado)
  • Nombre completo (obligatorio, mínimo 3 caracteres)
  • Email (obligatorio, formato válido, único)
  • Teléfono (obligatorio, formato español: 9 dígitos)
  • Fecha de registro (autogenerada)
  • Notas/preferencias (opcional)

3.2. Gestión de Mesas

Endpoint Método Descripción
/mesas/ GET Listar todas las mesas
/mesas/{id} GET Obtener una mesa por ID
/mesas/ POST Crear una nueva mesa
/mesas/{id} PUT Actualizar datos de una mesa
/mesas/{id} DELETE Eliminar una mesa (solo si no tiene reservas futuras)
/mesas/disponibles/ GET Listar mesas disponibles para una fecha/hora y número de comensales

Datos de la mesa:

  • ID (autogenerado)
  • Número de mesa (único, 1-99)
  • Capacidad (2, 4, 6 u 8 personas)
  • Ubicación (interior, terraza, privado)
  • Activa (boolean, para poder desactivar mesas temporalmente)

3.3. Gestión de Reservas

Endpoint Método Descripción
/reservas/ GET Listar reservas con filtros (fecha, cliente, mesa, estado)
/reservas/{id} GET Obtener una reserva por ID
/reservas/ POST Crear una nueva reserva
/reservas/{id} PUT Modificar una reserva existente
/reservas/{id} DELETE Cancelar una reserva
/reservas/{id}/confirmar PATCH Confirmar llegada del cliente
/reservas/{id}/completar PATCH Marcar reserva como completada

Datos de la reserva:

  • ID (autogenerado)
  • ID del cliente (obligatorio, debe existir)
  • ID de la mesa (obligatorio, debe existir y estar activa)
  • Fecha y hora de inicio (obligatorio, debe ser futura)
  • Fecha y hora de fin (calculada automáticamente: inicio + 2 horas)
  • Número de comensales (obligatorio, no puede exceder capacidad de la mesa)
  • Estado (pendiente, confirmada, completada, cancelada)
  • Notas especiales (opcional)
  • Fecha de creación (autogenerada)

3.4. Estadísticas

Endpoint Método Descripción
/estadisticas/ocupacion/diaria GET Ocupación por día (parámetro: fecha)
/estadisticas/ocupacion/semanal GET Ocupación de la semana (parámetro: fecha inicio)
/estadisticas/clientes-frecuentes GET Top 10 clientes con más reservas

4.3. Validaciones Requeridas

Validaciones de negocio que deben implementarse:

  1. No solapamiento de reservas: Una mesa no puede tener dos reservas que se superpongan en el tiempo.
  2. Capacidad de mesa: El número de comensales no puede exceder la capacidad de la mesa.
  3. Horario de operación: Las reservas solo pueden hacerse dentro del horario del restaurante.
  4. Reservas futuras: No se pueden crear reservas en el pasado.
  5. Mesa activa: Solo se pueden hacer reservas en mesas activas.
  6. Cliente existente: El cliente debe existir en la base de datos.
  7. Cancelación: Las reservas solo pueden cancelarse si están en estado "pendiente" o "confirmada".

4.4. Excepciones Personalizadas

Crea excepciones específicas para cada tipo de error:

class ClienteNoEncontradoError(Exception):
    """El cliente especificado no existe"""
    pass

class MesaNoDisponibleError(Exception):
    """La mesa no está disponible en el horario solicitado"""
    pass

class ReservaSolapadaError(Exception):
    """Ya existe una reserva para esa mesa en ese horario"""
    pass

class CapacidadExcedidaError(Exception):
    """El número de comensales excede la capacidad de la mesa"""
    pass

class FueraDeHorarioError(Exception):
    """La reserva está fuera del horario de operación"""
    pass

class CancelacionNoPermitidaError(Exception):
    """La reserva no puede cancelarse (muy tarde o estado inválido)"""
    pass

4.5. Base de Datos

Esquema de tablas SQLite:

-- Tabla de clientes
CREATE TABLE clientes (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    nombre TEXT NOT NULL,
    email TEXT UNIQUE NOT NULL,
    telefono TEXT NOT NULL,
    notas TEXT,
    fecha_registro TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- Tabla de mesas
CREATE TABLE mesas (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    numero INTEGER UNIQUE NOT NULL,
    capacidad INTEGER NOT NULL CHECK(capacidad IN (2, 4, 6, 8)),
    ubicacion TEXT NOT NULL CHECK(ubicacion IN ('interior', 'terraza', 'privado')),
    activa BOOLEAN DEFAULT 1
);

-- Tabla de reservas
CREATE TABLE reservas (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    cliente_id INTEGER NOT NULL,
    mesa_id INTEGER NOT NULL,
    fecha_hora_inicio TIMESTAMP NOT NULL,
    fecha_hora_fin TIMESTAMP NOT NULL,
    num_comensales INTEGER NOT NULL,
    estado TEXT DEFAULT 'pendiente' CHECK(estado IN ('pendiente', 'confirmada', 'completada', 'cancelada')),
    notas TEXT,
    fecha_creacion TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (cliente_id) REFERENCES clientes(id),
    FOREIGN KEY (mesa_id) REFERENCES mesas(id)
);

-- Índices para mejorar rendimiento
CREATE INDEX idx_reservas_fecha ON reservas(fecha_hora_inicio);
CREATE INDEX idx_reservas_mesa ON reservas(mesa_id);
CREATE INDEX idx_reservas_cliente ON reservas(cliente_id);

Request:

POST /reservas/
Content-Type: application/json

{
    "cliente_id": 1,
    "mesa_id": 5,
    "fecha_hora_inicio": "2024-12-20T20:00:00",
    "num_comensales": 4,
    "notas": "Cumpleaños, traer tarta"
}

Response (201 Created):

{
    "id": 21,
    "cliente_id": 1,
    "mesa_id": 5,
    "fecha_hora_inicio": "2024-12-20T20:00:00",
    "fecha_hora_fin": "2024-12-20T22:00:00",
    "num_comensales": 4,
    "estado": "pendiente",
    "notas": "Cumpleaños, traer tarta",
    "fecha_creacion": "2024-12-15T10:30:00",

        "nombre": "María García",

    "mesa": {
        "numero": 5,
        "ubicacion": "terraza"
    }
}

Crear una reserva (caso de error - solapamiento):

Request:

POST /reservas/
Content-Type: application/json

{
    "cliente_id": 2,
    "mesa_id": 5,
    "fecha_hora_inicio": "2024-12-20T21:00:00",
    "num_comensales": 2
}

Response (409 Conflict):

{
    "detail": {
        "error": "ReservaSolapadaError",
        "mensaje": "La mesa 5 ya tiene una reserva de 20:00 a 22:00 el día 2024-12-20",
        "sugerencia": "Mesas disponibles en ese horario: 3, 7, 12"
    }
}
